## 2. Reflection: First Idea vs Final Deliverable

My first idea was a very simple clicker game where the player would just tap water drops to increase a score. The final version kept that core mechanic, but evolved into a complete timed challenge with difficulty modes (Easy, Normal, Hard), dynamic hazards (pollution penalties), and a clearer game loop with win/loss states.

What stayed the same was the mission-focused concept: collecting water to represent support for clean water access. What changed was the depth of gameplay, user feedback, and polish. I added milestone messages, stronger visual hierarchy, on-brand styling, and clearer status updates so players always understand what is happening.

If I tackled a similar project again, I would prototype the game balance earlier and test with real users sooner. That would help me fine-tune score targets and pacing faster, and make the gameplay feel even smoother from the start.

## 3. Interview Question Response

### Prompt chosen: How do you approach debugging or troubleshooting in projects?

That is a great question. In my charity: water game project, I used a structured debugging process. First, I reproduced each issue consistently, like checking what happened when changing difficulty, clicking items quickly, or reaching the timer edge cases. Then I isolated the specific function involved, for example spawn timing, score updates, or win-state checks.

I relied heavily on small, focused tests after each change. Instead of rewriting big sections at once, I made one fix, validated behavior, and moved on. I also used clear UI status messages while developing, which made it easier to see when state transitions were wrong.

The biggest lesson was that gameplay bugs are often state-management bugs. Once I made game state explicit, such as tracking running mode, active timers, and milestone progress, troubleshooting became much faster and the game became more reliable.
